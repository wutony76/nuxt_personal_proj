## Context

目前 `app/pages/lottery/bg/6hc-cd.vue` 尚未具備任何信用盤玩法頁能力，僅有簡單標題。  
既有專案已完成官方盤 `6hc-of`，但信用盤玩法與操作節點尚未建立，導致無法對應 `pcv2_0223` 的 6HC 信用玩法清單進行驗收。

## Goals / Non-Goals

**Goals:**
- 在 `6hc-cd` 建立可擴充的玩法切換容器，並一次落地 14 個指定玩法。
- 將玩法清單集中在單一配置，避免模板硬編碼造成後續維護成本。
- 補齊 `openspec/specs/6hcCredit.md` 作為信用盤玩法來源與行為準則。

**Non-Goals:**
- 本次不實作完整下注送單 API 串接。
- 本次不重構 `6hc-of` 既有官方盤頁面與其組件結構。
- 本次不搬移 `pcv2_0223` 全量樣式與所有子組件，只保留玩法核心資訊與切換能力。

## Decisions

1. **以單頁內聚方式先完成玩法切換骨架**
   - 在 `6hc-cd.vue` 先完成玩法 tab 與內容區，確保頁面可直接測試全部玩法是否存在。
   - 相比立即拆分 14 個檔案，先集中可降低初次導入成本，後續再依複雜度抽離。

2. **玩法資料採單一 `reactive` state + 常量清單管理**
   - 依專案 state object 規範，以單一 `reactive` 物件維護當前玩法與展示資料。
   - 清單定義包含玩法 key、標題、描述，維持穩定排序並直接對應驗收項目。

3. **規格文件採 capability + page spec 雙軌**
   - 在 change 內提供 `specs/6hc-credit/spec.md` 做為此次能力變更記錄。
   - 在專案級 `openspec/specs/6hcCredit.md` 放最終玩法規格，供後續擴充（賠率、下注條件）沿用。

## Risks / Trade-offs

- [風險] 先以單頁骨架導入可能與未來下注模組拆分邊界不一致  
  → [緩解] 玩法資料結構預留 `groups` 欄位，後續可直接抽至 composable / component。

- [風險] 來源專案 `pcv2_0223` 與本專案命名可能有差異（例如 ixiaolian）  
  → [緩解] 以使用者指定 key 為主，規格與頁面顯示一致，避免隱性轉名。

- [取捨] 不一次搬入完整玩法表格與下注交互，先確保玩法覆蓋率  
  → [原因] 目前任務重點為玩法完整性與規格落地，下注流程可在後續 change 補齊。
